## Type System Expectations

- **Strict Typing:**
  - Use `type` (not `interface`) for props, state, and data models.
  - **Generated Types:** Always use types generated by GraphQL Code Generator for API data. Do not manually redefine API shapes.
  - **No `any`:** Avoid `any` (implicit or explicit); prefer `unknown` plus narrowing.
  - **Trust Your Types (No Defensive Coding in UI):**
    - **Validation at Boundary:** Validate data existence in the Mapper/Hook layer using Zod or Type Guards.
    - **Clean UI:** Inside a component, assume props are present if the type says so. Do not use `?.` everywhere "just in case".
    - **Fail Fast:** It is better to throw an error in a mapper than to render a broken UI silently.
- **React Integration:**
  - **Event Handlers:** Define explicit types for event handlers (e.g., `React.ChangeEvent<HTMLInputElement>`).
  - **Component Props:** Use `React.ComponentProps<T>` when wrapping native elements or other components to preserve standard props.
- **Generics & Unions:**
  - **Generics:** Use generics for reusable components and hooks.
  - **Union Types:** Use discriminated unions for state and events (e.g., `type State = { status: 'loading' } | { status: 'success', data: T }`).
- **Best Practices:**
  - Centralize shared contracts instead of duplicating shapes.
  - Express intent with TypeScript utility types (e.g., `Readonly`, `Partial`, `Record`).
- **Validation & Zod:**
  - **Schema Location:** Zod schemas should be located near the data they validate.
  - **Naming:** Use `*.schema.ts` or `schemas.ts`.
  - **Example:** `src/features/cart/cart.schema.ts` (Domain schemas), `src/mappers/cartMapper.schema.ts` (CMS validation schemas).
  - **Export:** Export the inferred type: `export type Cart = z.infer<typeof CartSchema>;`
